Introduction to Basic Availability Groups

Basic Availability Groups (BAGs) are a high-availability feature introduced with SQL Server 2016 Standard Edition. They are designed as a lightweight alternative to the more advanced Always On Availability Groups found in Enterprise Edition, primarily replacing the deprecated Database Mirroring feature. BAGs provide essential disaster recovery and failover capabilities for databases on SQL Server Standard, but with specific limitations tailored for smaller-scale or cost-sensitive environments.

Key Features

  • Support for a Single Database: Each Basic Availability Group can protect only one user database. This differs from advanced groups, which can include multiple databases.
  • Two Replica Limit: Only a primary (read-write) and a single secondary replica (standby) are allowed. This ensures a simple, two-server failover topology.
  • Synchronous & Asynchronous Modes: Replication between the primary and secondary replica supports both synchronous-commit (for high availability) and asynchronous-commit (for disaster recovery).
  • Automated Failover: An automatic failover can be configured between the two replicas, ensuring minimal manual intervention during server or database failure.
  • Seamless Integration: Configuration and management of BAGs use the same interfaces as advanced availability groups, including Transact-SQL (T-SQL) and SQL Server Management Studio (SSMS) from version 17.8 onward.

Major Limitations

  • No Readable Secondary: The secondary database is not accessible for read-only operations; all client queries must target the primary.
  • No Offloaded Backups: Backups and integrity checks cannot run on the secondary replica—these operations must occur on the primary.
  • No Upgrade Path: You cannot directly upgrade a BAG to an advanced Always On group. To move to Enterprise features, the BAG must be dropped and recreated as a new advanced group.
  • No Distributed Groups: BAGs cannot participate in distributed availability groups or complex, multi-replica topologies.
  • Standard Edition Only: Supported exclusively on SQL Server Standard Edition—not available for Enterprise Edition nodes as a supported feature.

Use Cases

Basic Availability Groups are most suitable for:

  • Small to Medium Businesses: Where Standard Edition is preferred for licensing cost.
  • Simple High-Availability Scenarios: Protection for a single database requiring minimal secondary server usage.
  • Legacy Mirroring Replacement: Environments transitioning from Database Mirroring and not needing complex, multi-database configurations.

How Basic Availability Groups Work

  • The primary replica hosts the active, read-write database.
  • The secondary replica maintains a synchronized, non-accessible copy. In case of a primary server failure, an automatic or manual failover promotes the secondary to become the new primary.
  • Data movement is handled over encrypted endpoints, usually via Windows Server Failover Clustering (WSFC).
  • The entire group is created using the T-SQL CREATE AVAILABILITY GROUP command with the WITH BASIC option. Example:
sql
CREATE AVAILABILITY GROUP [BasicAG]
WITH (BASIC)
FOR DATABASE [YourDatabase]
REPLICA ON ...
  • The secondary becomes active only during failover; there is no read access or reporting functionality on the secondary.

Practical Configuration Notes

  • Both the primary and secondary SQL Server instances must be SQL Server 2016 Standard Edition or later.
  • Always On must be enabled on both instances and nodes should be part of a Windows failover cluster.
  • Only one database per BAG is permitted.
  • Failover between the replicates can be configured as automatic (preferred for high availability) or manual (common in DR scenarios).

Conclusion

Basic Availability Groups provide a streamlined, cost-effective high-availability solution for SQL Server Standard Edition deployments, ideal for protecting a single critical database with automated failover between two nodes. While their feature set is drastically simplified compared to Enterprise Edition, they remain a valuable option for organizations with limited HA requirements or budgets, and for those phasing out Database Mirroring.
